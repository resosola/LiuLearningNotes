- Protobuf 的序列化和反序列化开销大吗？Protobuf 如何与共享内存结合使用？
  - Protobuf 的序列化性能在大多数场景下是可以接受的，但在高频通信中仍然存在一定开销。为了和共享内存结合使用，不能直接放入消息对象，而是通过 SerializeToArray() 的形式转为字节流写入，再使用 ParseFromArray() 恢复，确保数据结构的可移植性与跨进程安全性。
  - 序列化: 把对象转为二进制，字段是变长编码 有一定 CPU 消耗
  - 反序列化: 必须完整解析 tag + 类型 + value，有解析遍历的开销
  - Protobuf 消息对象是面向堆的结构体，内部可能包含指针（string、repeated 等）指针指向的是原进程堆区的地址，另一个进程无法访问这些地址

- 你们是如何进行进程间通信的？为什么选共享内存 + 环形缓冲区？
  - 共享内存是所有 IPC 方式中性能最高的，因为无系统调用开销、无复制、大吞吐量和延迟极低，共享内存是原始内存块，没有读写逻辑、没有边界管理，所以你必须在其上设计协议或结构，而环形缓冲区是最经典、最成熟的并发通信结构之一。
  - 用环形缓冲区是为了避免频繁 malloc/free，提高性能。
  - 使用分段读写，解决跨 buffer 边界的问题。
    - 分段读写的本质是把“连续的逻辑数据”映射到“物理上可能不连续的环形 buffer”中，确保完整性和正确性。
* 服务进程启动前 会依据配置文件创建指定shmKey和指定大小的共享内存，共享内存上写上预先定义的数据结构，即通道信息，通道信息指明 该通道 的发送进程 和 接受进程 各是什么，每条通道又会对半划分为两条子通道 一半用于接受数据  一半用于发送数据
* 服务启动时 进程会依据启动参数 得到shmKey并将共享内存 映射到 本地进程的虚拟内存中，然后读取共享内存信息 将发送端属于当前进程的通道信息 和 接收端属于当前进程的通道信息 存储到进程内存中
* 当进程需要发送数据 时 将数据写入 发送通道 中， 另一进程从 读取通道中 即可读取数据 
* 对通道内的数据基于环形缓冲区管理，通道读写位置 与 共享内存分离 由本地进程维护，数据读写遵循环形缓冲机制

* 通道实例结构：| 发送端id | 接收端id | 通道偏移值 | 通道大小 |
* 某个通道的实际数据结构： | 发送端 | 接收端 |

## SpuTun
技术栈：C++、Linux 策略路由、Linux网络编程、Linux多线程编程
项目描述：SpuTun 是为运行在 SPU内的进程提供网络通信能力的核心服务，服务基于 Linux TUN 与 PCIe 高速通道 实现，为机密计算提供安全高效的网络基础设施。

- ipv4：字符串 点分十进制  整数：32位无符号整数，A（最高字节）.B.C.D（最低字节）
- → (A << 24) | (B << 16) | (C << 8) | D

- Tun设备工作在ip层，其读写的是ip包，TUN 的上层是内核 IP 层协议栈，TUN 的下层是你自己的用户态程序（模拟网卡），内核IP层才会进行IP分片和重组，其负责把超大数据包拆分成符合 MTU 限制的小片（分片），以及在接收端重新组装这些分片（重组）。
- TUN 设备本身不负责 IP 分片和重组，它只是一个虚拟的网卡接口，向用户空间暴露“原始的 IP 包”。
- SPU应用通过Tun设备发包：
```bash
用户空间 HTTP Client
     ↓
  系统调用 send()
     ↓
Linux 内核 TCP/IP 协议栈处理：
     - TCP 分段
     - IP 构造（源地址 = tun0 的 IP）
     - 路由选择 dev = tun0
     ↓
输出 IP 包 → tun0 网卡 → 驱动 → /dev/net/tun
     ↓
用户态程序通过epoll从 /dev/net/tun 读取这些 IP 包  再通过pcie通道 发到宿主机中
```
- SPU应用收到回包：
```bash
从pcie通道拿到宿主机收到的回包 → write(tun_fd, ip包) → 内核以为tun0“收到了一个 IP 包”
                 ↓
          内核IP协议栈继续处理：
          - 路由
          - 防火墙
          - NAT
          - TCP/UDP 传输层
          - 最终交给应用进程
```
- TUN 设备内部的 buffer 是按 包为单位组织的（packet-based）。所以每次 read() 都只返回一个ip包

SPU是一款自研的硬件产品，类似一个小型的服务器，有自己的内存条、CPU、还有硬盘，有独立的操作系统和执行环境，SPU可以插在主板的pcie插槽上，并借用宿主机的供电运行，但是SPU自己是没有物理网卡的，所有它的网络通信是需要基于借助宿主机的物理网卡来进行，具体就是SPU会创建Tun 设备，其作用是提供一个点对点的虚拟网络接口，作用于IP层，供用户态程序发送和接收“IP 层的数据包”。Tun设备的IP将会和宿主机的物理网卡IP相同，
SPU内的网络请求包 会做以下处理：SPU内的某个进程发送某个应用层的请求包时，内核会将其对应的IP包 传给Tun设备，epoll从tun设备读取数据包，然后依据白名单进行出站流量管控，判断是否发行到宿主机，可放行则通过pcie通道 发往宿主机，宿主机再通过rawsocket 和sendmmsg() 将 该IP包转发至外部网络。
- epoll 和 select 都是 Linux 提供的 I/O 多路复用机制，用于高效地监听多个文件描述符（fd）是否可读/写/异常，常用于网络服务器和高并发程序中。
  - 项目	select	epoll
    系统调用接口	select(fd_set* readfds, ...)	epoll_create(), epoll_ctl(), epoll_wait()
    - select：select() 的 fd 集合在调用时会从用户态复制到内核态，内核对它们做线性遍历和状态检查，发现有就绪 fd 就返回，并将结果复制回用户态。
    - epoll_ctl()：fd 关联进 epoll 实例，内核维护红黑树记录所有监听 fd
    - epoll_wait(): 内核会将发生事件的 fd 放入 ready list（就绪队列），不再遍历所有 fd，而是直接返回就绪的 fd 列表，减少了大量无效扫描（这就是 O(1) 的本质）
    最大 fd 数限制	有限（默认 1024），需通过 FD_SETSIZE 控制	  无限制（上限是系统 max open files）
    - select 使用固定大小的位图 fd_set，最多只能监听 FD_SETSIZE（通常是 1024）个文件描述符，限制较明显；而 poll 使用可变长的 pollfd[] 数组，可以动态监听任意数量的 fd，仅受进程最大 fd 限制（如 ulimit -n），因此在大规模并发连接下更灵活。但两者本质上都采用线性遍历，性能为 O(n)，不适合大规模连接，生产推荐使用 epoll。
    fd 检测方式	每次调用都扫描全部 fd 集合	只检查真正就绪的 fd（事件驱动）
    性能（N个 fd）	O(N)，随 fd 数增加性能下降	O(1)，即使监听百万个 fd 也能高效
    内存使用	每次调用都需要复制 fd 集合到内核空间	内核保存 fd 列表，不重复传输
    触发方式	水平触发（level-triggered）	支持水平触发 和 边沿触发（edge-triggered）
    - 水平触发（level-triggered, LT） 模式：只要数据没读完，事件就会一直通知你。
    - 边沿触发（edge-triggered, ET） 模式：只有状态发生“变化”的那一刻才通知一次，比如从无数据到有数据。 【故事件触发后 需要 循环读/写 直到数据读完为止】
      模式	是否必须非阻塞	是否必须循环读	原因
      LT	  否	          否（可选）	    会重复触发，只要有数据就通知
      ET	  是	          是	            不会重复触发，不知道到底能读写多少数据 必须一次读干净 故需要循环读
      ET 适用于 成千上万个连接的场景	LT 模式可能每个都重复触发，epoll_wait 醒太频繁 	ET 只触发一次，避免重复 wakeup（节省系统调用）
      因为 我们的场景只有一个fd 即tun设备，且处理逻辑相对简单，ET 和 LT 没多大区别 故两种模式下性能差距没多大
      编程复杂度	低	高，容易漏事件
      性能（高并发下）	稍差	更高（减少 epoll_wait 唤醒次数）
    修改监听集	每次调用都要重新设置	epoll_ctl 添加/删除，监听集在内核中维护
    跨平台支持	POSIX 标准，几乎所有 Unix/Linux 支持	仅 Linux 支持（内核 ≥ 2.6）
    编程复杂度	简单但低效	稍复杂，但性能优越
    默认行为	    阻塞	阻塞
    控制阻塞方式	设置 timeout	设置 timeout
    可非阻塞	    ✅ 是的	✅ 是的
    是否影响线程调度	✅ 阻塞调用线程	✅ 阻塞调用线程
  - select/epoll 的本质是“事件通知 + 阻塞等待” 默认行为是阻塞线程直到事件到来，超时和非阻塞是可配置的，靠参数决定，select、epoll_wait都需阻塞
  - epoll多线程架构【单生产者单消费者】：
    - 循环等待事件触发线程：该线程从TUN设备中读取网络包 放入队列
      - 使用epoll_wait监听TUN设备的可读事件
      - 每当可读时，将读取到的数据封装成对应数据结构，压入队列中
    - 消费线程：从队列中取出数据包并回调处理
    - 队列使用互斥锁 吞吐量（总字节数 / 总时间（秒））较低
      - 性能限制：
        - 线程之间频繁锁竞争（尤其是生产者/消费者频繁交替时）
        - wakeup 机制（notify_one()）触发上下文切换，会带来 非微不足道的延迟
        - 整个队列是单锁结构：enqueue / dequeue 无法并行
    - 可考虑使用无锁 ring buffer实现 提高吞吐量 无锁、无依赖、极快  平均读吞吐量提升约 4.8 倍（≈379% 提升）
  - 常见线程通信模型概览：
  模型名称	生产者线程数	消费者线程数	是否加锁	场景示例
  SPSC（Single Producer, Single Consumer）	1	1	❌ 无锁可实现	网络驱动线程 ➝ 单线程逻辑处理
  SPMC（Single Producer, Multi Consumer）	1	多	✅ 需加锁或无锁结构	一线程收数据 ➝ 多线程并发处理
  MPSC（Multi Producer, Single Consumer）	多	1	✅	多线程采集数据 ➝ 主线程写数据库
  MPMC（Multi Producer, Multi Consumer）	多	多	✅✅ 必须加锁或用成熟无锁库	高并发任务队列、多线程分发

    - 消费者：谁在处理这些数据？

- **I/O 多路复用主要监听可读事件 (readable) 是主流用法，但写事件 (writable) 也非常重要，**只是写事件一般是“按需注册”，不常驻监听。
   - 对于大多数 socket，只要发送缓冲区有空位，就被认为是可写的。所以只有在你真的遇到 "写不进去" 的情况时，才需要临时监听写事件。
- RAW Socket 是一种套接字类型，它允许应用程序直接收发网络层数据包，而不是通过内核自动封装的 TCP 或 UDP 连接。
- sendmmsg() 批量通过 Raw Socket 发送多个数据包，可以大幅减少系统调用开销、提高发送性能（PPS），特别适合发送大量小包
SPU收到的网络响应包 会做以下处理：宿主机收到入站流量后，经白名单过滤后，通过 PCIe 通道注入 SPU 内，epoll写入IP包到tun设备中，由目的端口匹配的源进程读取处理。




主要内容：
基于 epoll 事件驱动 + 多线程架构的 TUN 设备管理模块，实现了高效、可靠的虚拟网络设备管理，为 SPU 系统提供了高性能通信能力。
通过 iptables 规则链 将宿主机指定网口的入站流量重定向至 NFQUEUE，在用户态基于 Netfilter Queue (NFQ) 注册回调函数捕获原始数据包，经白名单过滤后，通过 PCIe 通道 将合法流量注入 SPU 内部处理。
* 将指定网络接口 ($iface) 接收到的所有入站数据包，转发到 NFQUEUE（Netfilter Queue）指定队列号 的队列中，供用户态程序处理。NFQUEUE（Netfilter Queue）是 Linux 内核中的 Netfilter 子系统提供的一个 机制，它允许你把数据包从内核转发到用户态程序，把网络包交给用户空间程序处理，并等你决定是否放行、修改或丢弃它。
实现 基于IP/端口的白名单过滤引擎，拒绝非法流量出入SPU，实现精细化流量管控。
通过 RAW Socket 构造原始数据包，将 SPU 处理后的流量重新注入宿主机内核协议栈，确保数据包按标准网络路径转发至外部网络或本地应用。
- Raw Socket 的上层是用户态程序，底层是内核网络层协议（IP 层）。
```bash

   ↑  ↓
 Raw Socket (用户态创建的 socket)
   ↑  ↓

 ┌───────────────┐

 │    TCP/UDP    │ ← 跳过/不参与（你构造的原始包可不用它）
 └───────────────┘
   ↑  ↓
网络驱动 / 网卡
```

## BluePrintPackage

  - 通过CMake获取运行的系统平台：Mac或者Windows来进行配置，Mac平台下再依据对应的处理器架构（X86或者Arm架构)进行对应依赖库的配置，mac下需配置为Clang编译器，windows下需配置MSVC编译器
  - 编译选项
- 基于OOP的算法集成框架的设计，统一并规范数据输入、处理、输出流程，保证代码结构简洁、可拓展性高、提高开发效率。
  - 前端传入本地服务的数据格式是基于protobuf序列化协议，对于每个接口有对应的proto格式文件
  - 每个功能接口都有对应的功能处理类，其派生自同一个功能基类，功能基类为模板类，其模板实参为对应功能参数，功能基类的初始化方法可依据模板实参进行proto串读取解析，并抽出数据用于实例化对应参数类对象
  - 功能处理类对应处理接口通过虚函数机制覆盖了基类的处理接口，达到多态的运行效果，处理接口主要进行对应功能模块的逻辑处理并进行响应数据的封装

  - 功能处理依赖对应算法接口，故算法接口内部发生错误到响应给前端时，状态码是从算法状态码到集成状态码再到前端状态码的转换过程。

主要内容：

- 服务端执行listen后 内核会维护两个连接队列 一个半连接队列，一个全连接队列
- 半连接队列：服务端收到SYN包后将连接加入此队列，等待三次握手完成
- 全连接队列；三次握手完成的连接放到此队列中
- 服务端 accept 调用后的返回 是 全连接队列队列中 取出一个连接
  - 若队列为空，则阻塞（默认行为）；
  - 若队列非空，返回一个新的 socket fd（client_fd），这个 fd 可以读写数据。
